📚 NODUS Supabase Migration Tutorial

Un guide completo explicando cómo el frontend de NODUS cambió de datos hardcodeados a una base de datos Supabase PostgreSQL en vivo.

🎯 Visión General

La migración de NODUS transforma la aplicación de un modelo estático a uno dinámico:

Antes: Todos los datos estaban en archivos TypeScript (500+ líneas de nodos, 1000+ líneas de contenido).

Después: Los datos viven en Supabase PostgreSQL con acceso seguro a través de APIs REST.

❌ El Problema (Antes)

Arquitectura Original

NODUS almacenaba todo en archivos estáticos:

// lib/graph-data.ts const nodeData = [ { id: "calc1", label: "Cálculo I", ring: 1, angle: 30, ... }, { id: "calc2", label: "Cálculo II", ring: 2, angle: 45, ... }, // ...cientos de líneas más ] // lib/node-content.ts const nodeContent = { calc1: { summary: "...", concepts: [...], why: "...", example: "...", difficulty_note: "..." }, // ...más contenido }

Problemas Identificados

  • Contenido estático: Cualquier cambio requería un deploy de código
  • Sin persistencia: Las nuevas contribuciones se perdían
  • Escalabilidad: Archivos grandes ralentizan el bundle
  • Sin dinámicas: Imposible distinguir contenido publicado vs borrador
  • Errores propensos: Editar objetos TypeScript manualmente genera bugs

✅ La Solución (Después)

Nueva Arquitectura: Supabase PostgreSQL

┌─────────────────────┐ │ Navegador / App │ │ (Lado del Cliente) │ └──────────┬──────────┘ │ NEXT_PUBLIC_SUPABASE_ANON_KEY │ (Pública, segura de commitear) ▼ ┌──────────────────────────────────┐ │ Supabase PostgreSQL Database │ │ ┌────────────────────────────┐ │ │ │ nodes table │ │ RLS: SELECT público cuando estado='published' │ ├────────────────────────────┤ │ │ │ node_content table │ │ RLS: SELECT público permitido │ ├────────────────────────────┤ │ │ │ node_edges table │ │ RLS: SELECT público permitido │ └────────────────────────────┘ │ └──────────────────────────────────┘
✓ Beneficios Inmediatos: Contenido dinámico, datos persistentes, seguridad integrada, escalabilidad, sin deployes para cambios de contenido.

🔑 Cambios Clave Explicados

1️⃣ Cliente Supabase (lib/supabase.ts)

¿Qué es? Un wrapper que inicializa el cliente JavaScript de Supabase.

import { createClient } from "@supabase/supabase-js" const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL const supabaseAnonKey = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY if (!supabaseUrl || !supabaseAnonKey) { throw new Error("Faltan variables de entorno de Supabase") } export const supabase = createClient(supabaseUrl, supabaseAnonKey)

¿Por qué?

  • Usa variables de entorno en lugar de hardcodear URLs
  • El prefijo NEXT_PUBLIC_ hace que sea seguro para el cliente
  • Fuente única de verdad para la conexión Supabase

2️⃣ Capa de Datos (lib/data.ts)

¿Qué es? Funciones async que buscan datos de las tablas Supabase.

Función: fetchNodeData()

Obtiene todos los nodos publicados con sus relaciones de prerequisitos.

export async function fetchNodeData(): Promise<NodeDatum[]> { // 1. Obtener todos los nodos publicados const { data: nodes } = await supabase .from("nodes") .select("*") .eq("estado", "published") // Solo nodos publicados // 2. Obtener las relaciones de prerequisitos const { data: edges } = await supabase .from("node_edges") .select("*") // 3. Construir mapa de prerequisites desde edges const prereqMap = new Map<string, string[]>() edges?.forEach((edge: any) => { if (!prereqMap.has(edge.target_id)) { prereqMap.set(edge.target_id, []) } prereqMap.get(edge.target_id)!.push(edge.source_id) }) // 4. Retornar nodos formateados con prerequisites return nodes.map((n: any) => ({ id: n.id, label: n.label, desc: n.description, school: n.school, ring: n.ring, angle: n.angle, diff: n.difficulty || 0, prereqs: prereqMap.get(n.id) || [], })) }

¿Por qué este enfoque?

  • Filtrado de datos: Solo nodos publicados son visibles a usuarios anónimos
  • Cache: Los resultados se guardan para evitar solicitudes repetidas
  • Datos estructurados: Los prerequisites se buscan por separado y se fusionan para eficiencia

Función: getNodeContent(nodeId)

export async function getNodeContent(nodeId: string): Promise<NodeContent | null> { const { data } = await supabase .from("node_content") .select("*") .eq("node_id", nodeId) .single() // Espera exactamente una fila return { summary: data.summary, concepts: data.concepts || [], why: data.why, example: data.example, difficulty_note: data.difficulty_note, } }

¿Por qué: Separa el contenido (a menudo grande) de los metadatos del nodo, busca solo cuando es necesario, y usa .single() para seguridad de tipos.

3️⃣ Componentes Actualizados

components/nodus/knowledge-graph.tsx

Antes ❌

import { nodeData } from "@/lib/graph-data" export function KnowledgeGraph() { // Usa nodeData directamente }

Después ✓

import { fetchNodeData } from "@/lib/data" export function KnowledgeGraph() { const [nodes, setNodes] = useState([]) const [isLoading, setIsLoading] = useState(true) useEffect(() => { const load = async () => { setIsLoading(true) const data = await fetchNodeData() setNodes(data) setIsLoading(false) } load() }, []) if (isLoading) { return <p>Cargando...</p> } // Pasar nodos al hook useGraphEngine(isDark, nodes, ...) }

¿Por qué estos cambios?

  • Estado de carga: Los usuarios ven retroalimentación mientras los datos se cargan
  • Inyección de dependencias: Los nodos se pasan como parámetro, no se importan directamente
  • Testeable: El hook puede testearse con nodos mock

hooks/use-graph-engine.ts - Firma Cambiada

Antes ❌

useGraphEngine(isDark, onNodeClick?, activeSchool?)

Después ✓

useGraphEngine(isDark, nodes, onNodeClick?, activeSchool?, ...)

¿Por qué:

  • Separación de responsabilidades: El hook no sabe de dónde vienen los datos
  • Flexibilidad: Puede pasarse diferentes fuentes de datos
  • Testeable: Fácil de testear con configuraciones específicas

4️⃣ Páginas del Servidor

app/nodo/[id]/page.tsx

Antes ❌

export function generateStaticParams() { return nodeData.map(n => ({ id: n.id })) } export default function NodePage() { const node = nodeData.find(...) const content = nodeContent[id] }

Después ✓

// Sin generateStaticParams export default async function NodePage() { const nodes = await fetchNodeData() const node = nodes.find(...) const content = await getNodeContent(id) }

¿Por qué:

  • Renderizado bajo demanda: Los nuevos nodos de Supabase están inmediatamente disponibles
  • Sin generación estática: Evita páginas en caché desactualizado
  • Actualizaciones en tiempo real: Los cambios de contenido se reflejan instantáneamente

5️⃣ Script de Seed (scripts/seed.ts)

Propósito: Inicializar las tablas Supabase con datos de los archivos hardcodeados.

const supabaseUrl = process.env.SUPABASE_URL const supabaseSecretKey = process.env.SUPABASE_SECRET_KEY if (!supabaseUrl || !supabaseSecretKey) { console.error("Faltan variables de entorno") process.exit(1) } // Usar SECRET_KEY (privilegios de admin) const supabase = createClient(supabaseUrl, supabaseSecretKey) // 1. Leer lib/graph-data.ts // 2. Insertar nodos en tabla nodes con estado: "published" // 3. Insertar relaciones en tabla node_edges

¿Por qué este enfoque?

  • Usa SUPABASE_SECRET_KEY (admin) en lugar de la clave ANON
  • Nunca hardcodeado — solo desde .env.local
  • Operación única para bootstrapear la base de datos

Cómo ejecutar:

SUPABASE_URL=https://... SUPABASE_SECRET_KEY=sb_secret_... npx ts-node scripts/seed.ts

🔒 Modelo de Seguridad

Tres Conceptos Clave

1. Claves Públicas (Seguras de commitear)

NEXT_PUBLIC_SUPABASE_URL=https://rfxjdgmfohgnwsxrtzih.supabase.co NEXT_PUBLIC_SUPABASE_ANON_KEY=sb_publishable_jzDp_t0MD0RWbtbEThPDjQ_...
  • Visibles en las solicitudes de red del navegador
  • Acceso de solo lectura al contenido publicado
  • Usado por todos los clientes

2. Clave Secreta (¡NUNCA commitear!)

SUPABASE_SECRET_KEY=sb_secret_rfxjdgmfohgnwsxrtzih_...
  • Privilegios completos de admin
  • Usado SOLO en .env.local y scripts de servidor
  • Bloqueado por GitHub secret scanning

3. Políticas Row Level Security (RLS)

-- Cualquiera puede SELECT nodos publicados CREATE POLICY public_published_nodes ON nodes FOR SELECT USING (estado = 'published') -- Cualquiera puede SELECT contenido CREATE POLICY public_content ON node_content FOR SELECT USING (true)
  • Enforcement a nivel de base de datos
  • Sin política = sin acceso (default deny)
  • La clave ANON solo puede leer; nunca modificar
⚠️ Importante: GitHub secret scanning bloqueará commits que contengan claves secretas. Todas las credenciales deben estar en .env.local, nunca en archivos comprometidos.

🌍 Variables de Entorno

Archivos Requeridos

.env.local (¡Nunca commitear!)

📊 Flujo de Datos

Cuando un usuario visita /nodo/calc1:

1. El navegador solicita https://nodus-beta.vercel.app/nodo/calc1
↓
2. Servidor Next.js (Node.js runtime en Vercel)
  • Llama fetchNodeData()
  • └─ Usa NEXT_PUBLIC_SUPABASE_ANON_KEY
  • └─ Consulta: SELECT * FROM nodes WHERE estado='published'
  • └─ Retorna ~92 nodos
  • Encuentra nodo con id="calc1"
  • Llama getNodeContent("calc1")
  • └─ Consulta: SELECT * FROM node_content WHERE node_id='calc1'
  • └─ Retorna: summary, concepts, examples, etc.
↓
3. Renderiza HTML con contenido
↓
4. El navegador recibe la página pre-renderizada (sin estado de carga necesario)

Cuando el usuario abre el grafo (/grafo):

1. El navegador carga /grafo (componente cliente)
↓
2. React monta KnowledgeGraph
  • Muestra "Cargando grafo..." (estado de carga)
  • useEffect llama fetchNodeData()
  • └─ Fetch del lado del navegador con ANON_KEY
  • └─ Retorna ~92 nodos
  • Llama useGraphEngine(isDark, nodes, ...)
  • └─ Canvas renderiza con nodos, bordes, interacciones
↓
3. Grafo completamente interactivo

📝 Resumen

La migración de NODUS reemplaza archivos estáticos hardcodeados con un backend Supabase dinámico:

  1. lib/supabase.ts — Inicialización del cliente
  2. lib/data.ts — Búsqueda async de datos con consultas Supabase
  3. Componentes — Buscan datos en mount, muestran estados de carga, pasan datos vía inyección de dependencias
  4. Páginas del servidor — Renderizado async con datos buscados en el servidor
  5. Seguridad — Políticas RLS, claves secretas solo en ambiente, sin credenciales hardcodeadas
Resultado: Una plataforma de grafo de conocimiento en vivo, dinámica, segura y escalable.
